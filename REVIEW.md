# Code Review: IBM mcp-context-forge как кандидат на роль gateway для Tableau MCP в изолированном гос-контуре

Контекст ревью — `ARCHITECTURE.md` в этом репозитории, раздел 3 ("Вариант 2 — на базе IBM ContextForge") и связанные с ним разделы 5–10. Ревью кода, не тестов и не веб-морды. Цель — ответить, насколько ContextForge применим, безопасен и работоспособен именно в условиях изолированного контура серьёзной государственной организации, и закрыть открытые вопросы раздела 10 фактами из кода, а не из документации проекта.

Версия кодовой базы: снэпшот в `/home/egor/dev/ibm/mcp-context-forge` на момент ревью (2026-06-18).

---

## TL;DR

ContextForge — зрелый, активно поддерживаемый gateway с честной попыткой закрыть ровно тот контур задач, что описан в `ARCHITECTURE.md`: федерация tools, namespacing, OAuth2.1, передача токена в апстрим, observability, multi-cluster через Redis, авгаренный (air-gap aware) сборочный процесс. Но он **не реализует ту PAT-модель, которую `ARCHITECTURE.md` называет основной (раздел 6 — "enrollment + vault keyed by sub, PAT никогда не идёт по сети")**. Фактическая модель ContextForge — это раздел 7 "Путь 2" (PAT/токен передаётся клиентом в заголовке на каждый вызов, gateway форвардит его как `Authorization` в апстрим). Это не баг, а архитектурное решение продукта, и сам `ARCHITECTURE.md` в разделе 3 интуитивно это предвидел ("ContextForge решает per-user... через forwarding токена в заголовке на каждый вызов") — ревью кода это подтверждает на 100%.

Для гос-контура с требованием изоляции и отсутствия исходящего трафика: сборка и runtime **спроектированы с осознанием air-gap** (явный флаг `MCPGATEWAY_UI_AIRGAPPED`, скрипт вендоринга CDN-ассетов, отключённая по умолчанию телеметрия/OTEL/SIEM), но плагин-фреймворк **не является security boundary** — это нужно учитывать при решении, что вообще размещать в виде плагинов, а что — выносить в отдельный обвязанный процесс.

## 1. Применимость к описанной архитектуре (разделы 2–6 ARCHITECTURE.md)

### 1.1 Tool federation, namespacing, единый `/mcp`
Реализовано и рабочее: gateway регистрирует upstream MCP-серверы (включая stdio через встроенный `mcpgateway.translate`), агрегирует `tools/list` в один каталог, поддерживает `listChanged`. Это закрывает раздел 2 и идею "list один раз, invoke на пользователя" из раздела 9 — она там предусмотрена как часть архитектуры самого ContextForge, а не как наша доработка.

### 1.2 Per-user identity → per-upstream-session изоляция (раздел 5–6, "главное решение")
Здесь нашлось ключевое и обнадёживающее: `mcpgateway/services/upstream_session_registry.py` реализует **структурную изоляцию upstream-сессий по ключу `(downstream_session_id, gateway_id)`**, а не по identity/URL, как было раньше (это фикс конкретного issue про утечку состояния между чатами одного пользователя). Это значит: ContextForge умеет держать отдельное upstream-подключение на сессию, с собственным lifecycle-task и корректным cleanup при разрыве. Это закрывает Вариант A из раздела 5 ("процесс/сессия на пользователя") **на уровне HTTP/SSE upstream**.

Важная оговорка: это работает, если Tableau MCP выставлен как HTTP/SSE upstream. Если Tableau MCP оборачивается через `mcpgateway.translate` как stdio→HTTP мост (как и предполагает раздел 3 ARCHITECTURE.md: "stdio — оборачивается утилитой `mcpgateway.translate`"), то сам `translate.py` в режиме одного запущенного моста держит **один shared stdio-процесс** на весь мост (`mcpgateway/translate.py`, `StdIOEndpoint`/`_PubSub`), а не процесс на пользователя. То есть нужна одна из двух схем:
- регистрировать в ContextForge **по одному `translate`-мосту на пользователя** (тогда per-session изоляция registry бесполезна, изоляция обеспечивается на уровне процессов моста, что близко к Варианту A из раздела 5, только размазано по двум слоям — это работает, но усложняет lifecycle/TTL-эвикцию, которую тогда придётся реализовывать самим, а не получить из коробки);
- либо запускать собственный пул per-user мостов сбоку (гибрид, который `ARCHITECTURE.md` в разделе 3 уже называет как риск: "это уже гибрид: ContextForge как фронт + наш собственный sidecar-флот").

**Вывод по открытому вопросу №1 из раздела 10**: ContextForge закрывает per-session изоляцию на уровне HTTP/SSE upstream "из коробки", но не закрывает её бесшовно для stdio-апстрима с динамическим per-user PAT — там придётся либо переписывать апстрим Tableau MCP под per-request auth (что и сам документ называет условием №1 в разделе 3), либо городить sidecar-флот мостов (условие №2). Рекомендация ARCHITECTURE.md "начинать с эксперимента" — правильная и подтверждается кодом: без эксперимента над конкретной связкой "translate + Tableau MCP + multi-user" решение не однозначно.

### 1.3 Forwarding токена в апстрим (`X-Upstream-Authorization`)
Подтверждено в коде (`mcpgateway/utils/passthrough_headers.py:218-244, 357-368, 716-778`): заголовок `X-Upstream-Authorization` клиента переименовывается в `Authorization` и форвардится в апстрим, с санитизацией от CRLF/control-символов, но **без проверки, что значение принадлежит текущей identity** — это просто passthrough значения, которое прислал клиент в текущем запросе. Технически это ближе к "сырому PAT в заголовке" (раздел 7, Путь 2 / `mcp-remote`), а не к модели enrollment-в-vault из раздела 6, где "PAT никогда не отправляется заголовком на каждый запрос".

Есть `plugins/vault/vault_plugin.py` — единственный кандидат на роль раздела 6 ("vault, ключ = sub"). При детальном чтении (`tool_pre_invoke`, `_parse_vault_token_key`) выясняется: плагин транслирует заголовок `X-Vault-Tokens`, который **клиент также передаёт сам, на каждый запрос** (просто в более структурированном JSON-формате "система → токен"), и подставляет соответствующий bearer в апстрим. Хранилища "зашифрованный PAT по ключу sub, который сервер вынимает сам после OAuth-логина" в коде нет. То есть ни встроенный passthrough, ни плагин `vault` не реализуют дизайн из раздела 6 в чистом виде — там нужна доработка (либо собственный плагин/слой, который на этапе OAuth-enrollment кладёт PAT в БД по `sub` и сам инжектит его в `X-Upstream-Authorization` перед вызовом upstream, не пропуская сырой токен через клиента вообще).

### 1.4 Хранение секретов (PAT) at rest
В БД (`mcpgateway/db.py`, миграция `a706a3320c56_use_argon2id_for_encryption_key.py`) есть инфраструктура шифрования: Argon2id для деривации ключа из `AUTH_ENCRYPTION_SECRET`, Fernet для симметричного шифрования хранимых секретов/токенов. Это закрывает требование раздела 6 "секреты не логируются... лежат только в зашифрованном vault". Но: ключ шифрования — **один глобальный `AUTH_ENCRYPTION_SECRET`** на всю инсталляцию, без per-user envelope-ключей и без встроенной поддержки внешнего KMS/HSM. Для серьёзного гос-контура это стандартная практика (один master secret, обычно из HSM/секрет-менеджера на уровне инфраструктуры), но нужно явно завести `AUTH_ENCRYPTION_SECRET` через корпоративный секрет-менеджмент (Vault/KMS), а не как plain env var в compose-файле — в коде это просто `Field`, политика хранения — целиком на стороне деплоя.

### 1.5 Redaction секретов в логах
Частично закрыто: есть фильтр на редакцию query-параметров в httpx-логах (`mcpgateway/services/logging_service.py`) и `trace_redaction.py`. Явного централизованного "маскируй `Authorization`/`X-Upstream-Authorization` во всех логах и трейсах" не обнаружено как единой гарантии фреймворка — это разрозненные точечные фильтры. **Перед продом надо отдельно аудитировать**: что попадает в access-логи nginx/edge, в request-логи самого FastAPI при `DEBUG`, и в текст ошибок при исключениях апстрима (раздел 6 ARCHITECTURE.md явно требует "никогда — в логах, URL'ах и текстах ошибок", и это требует ручной проверки, а не считается решённым по умолчанию).

### 1.6 Session manager / session fixation
Сессии (`mcpgateway/cache/session_registry.py`) привязывают `session_id → owner_email`, ключ — UUID. В Redis-backend `claim_session_owner()` не строго атомарен (GET, потом SET под локальным asyncio-lock, который не защищает от другой реплики/инстанса) — при распределённом деплое (что нужно для multi-cluster, см. раздел 9 ARCHITECTURE.md) есть теоретическое окно race condition при одновременном claim одного `session_id` двумя инстансами. Риск невысокий практически (session_id — непредсказуемый UUID, не атакуется без побочного канала), но если делаете session affinity на балансировщике (как и рекомендует раздел 9), этот риск дополнительно снижается естественным образом — почти весь трафик одной сессии и так идёт на одну реплику.

## 2. Применимость к Claude Desktop / fallback на локальные модели (разделы 7–8)

Код это не блокирует и не противоречит документу: OAuth 2.1 resource server поведение (401 + `WWW-Authenticate` + Protected Resource Metadata) — это стандартная часть FastAPI MCP-роутера ContextForge, путь 1 из раздела 7 реализуем без правок ядра. Путь 2 (stdio-мост) у ContextForge есть "из коробки" как `mcpgateway.wrapper` — то, что архитектурный документ и сам указывает как альтернативу `mcp-remote`. Раздел 8 (фолбэк на локальные модели) ни на что в коде не давит — это просто следствие того, что снаружи стандартный MCP-сервер; ничего дополнительного в ContextForge для этого не нужно.

## 3. Применимость к изолированному (air-gapped) гос-контуру

Это была отдельная и самая важная для вас линия ревью, ниже — выжимка.

### 3.1 Исходящий трафик в runtime — по умолчанию отсутствует
- OpenTelemetry выключен по умолчанию (`mcpgateway/config.py`: `otel_enable_observability: bool = Field(default=False, ...)`), `observability.py` явно ранний `return None`, если флаг не включён.
- SIEM-экспорт (Splunk/Datadog/Elasticsearch/Webhook/Syslog) выключен по умолчанию (`siem_export_enabled: bool = Field(default=False, ...)`).
- SSO (Keycloak/Okta/Auth0/Google/GitHub/Entra/ADFS) — каждый провайдер за отдельным `sso_<provider>_enabled`, по умолчанию `false`.
- Нет встроенных SDK телеметрии использования (Sentry/PostHog/Mixpanel/Amplitude/New Relic) ни в зависимостях, ни в коде — не найдено ни одного обращения.
- `docker-compose.with-phoenix.yml`, `docker-compose.with-langfuse.yml`, `docker-compose.siem-opensearch.yml`, `docker-compose.sso.yml` — это всё отдельные опциональные compose-профили для локальных (self-hosted) интеграций, не часть дефолтного стека и не обращаются за периметр сами по себе.

**Вывод**: проект и команда явно проектировали ContextForge как "тихий" по умолчанию — это сильный плюс для гос-контура, такое нечасто встречается в проектах с богатой observability-историей.

### 3.2 Admin UI и CDN — единственная реальная находка, требующая действия
В `templates/login.html` и `templates/admin.html` по умолчанию подключаются внешние CDN: `cdn.tailwindcss.com`, `cdnjs.cloudflare.com` (Font Awesome, CodeMirror), `cdn.jsdelivr.net` (Chart.js, marked, DOMPurify). В air-gap контуре без egress-allowlist на эти домены Admin UI частично или полностью сломается визуально/функционально. Решение есть и оно реальное, не косметическое: флаг `MCPGATEWAY_UI_AIRGAPPED=true` (`mcpgateway/config.py`) плюс скрипт `scripts/download-cdn-assets.sh`, который вендорит те же версии библиотек локально в `mcpgateway/static/vendor/` на этапе сборки образа. **Это нужно включать явно и проверять на этапе сборки/приёмки**, дефолт — внешний CDN.

### 3.3 Supply chain / сборка образа без интернета
- Лицензия проекта — Apache-2.0, политика зависимостей (`license-policy.toml`, `deny.toml`) явно запрещает GPL/LGPL/AGPL/SSPL в продуктовых зависимостях (исключения только для dev-тулинга, не идут в runtime-образ) — юридически чисто для коммерческого/гос. использования.
- Python: ~113 прямых, ~377 транзитивных зависимостей (`uv.lock`) — придётся полностью зеркалировать в внутренний PyPI-индекс, это нормальный объём для FastAPI-проекта такого размера, но требует процесса.
- Node.js нужен только на этапе сборки фронтенда (Tailwind/Vite), в runtime-образе бинарники не нужны — собранная статика коммитится/собирается в образ.
- Опциональные Rust-расширения (`crates/`) выключены по умолчанию (`ENABLE_RUST=false`); если включать — требуют `rustup` (скачивается с `sh.rustup.rs`) и компиляции под целевую архитектуру. Для air-gap самый чистый путь — собирать Rust-часть на отдельном build-раннере с доступом в интернет (или зеркалом crates.io) и поставлять прероизведённые wheels, что в Containerfile.lite уже частично предусмотрено (stage для s390x pre-built wheels с GHCR).
- Базовые образы — `registry.access.redhat.com/ubi10/*` (Red Hat UBI), а Helm `values.yaml` по умолчанию указывает на публичный `ghcr.io/ibm/mcp-context-forge` — оба переопределяемы на внутренний registry, но по умолчанию настроены на внешние источники, это типовая, ожидаемая работа для любого деплоя в закрытый контур, а не специфический изъян проекта.

**Вывод**: air-gapped сборка реалистична и не "тяжело", если воспринимать её как обычный процесс зеркалирования (PyPI + npm devDependencies + UBI10 base + опционально crates.io), но это процесс, который нужно построить заранее, а не "из коробки по умолчанию".

## 4. Плагин-фреймворк — главный архитектурный риск, если туда класть кастомную бизнес-логику

`ARCHITECTURE.md` (раздел 3) рассматривает плагины ContextForge как один из двух способов разместить "кастомные инструменты". По факту кода это требует прямой оговорки:

- Плагины загружаются по полному пути класса (`kind: "plugins.module.ClassName"`) из YAML-конфига и **импортируются динамически без whitelist**`mcpgateway/plugins/gateway_plugin_manager.py`. Если конфиг попадёт под контроль атакующего (компрометация репозитория конфигурации, CI/CD, неверные file permissions) — это прямой путь к выполнению произвольного Python-кода в процессе gateway.
- Плагины выполняются **in-process**, без sandbox: тот же интерпретатор, та же память, тот же доступ к `os.environ`, файловой системе, сети, БД, что и у самого gateway. Хуки `tool_pre_invoke`/`http_pre_request` отдают плагину **все заголовки запроса**, включая `Authorization`/`X-Upstream-Authorization`/PAT.
- Никакой permission/scope-модели на уровне фреймворка нет: манифест плагина (`plugins/vault/plugin-manifest.yaml`) — это просто описание, не декларация прав. Нет встроенного способа ограничить плагину доступ к сети/диску/БД.
- Качество примеров неоднородное: `header_filter`, `header_injector`, `vault` выглядят продуктово готовыми; `safe_html_sanitizer` — хрупкий regex-фолбэк вместо полноценного HTML-парсера; `harmful_content_detector` — игрушечный demo с хардкод-словарём, не для реального использования as-is.

**Вывод**: плагин-фреймворк ContextForge — это механизм для **доверенного, проверенного силами вашей команды кода**, не security boundary между разными уровнями доверия. Для серьёзного гос-контура это означает:
- весь код плагинов проходит тот же ревью/CI/подпись, что и сам gateway — плагин-конфиг должен быть частью защищённого релизного пайплайна, а не редактируемым через Admin UI без контроля;
- если по архитектуре нужна изоляция между разными доверительными зонами (например, кастомная Tableau-логика и сторонние тулы) — её нужно строить вне плагин-фреймворка: отдельный MCP-сервер/процесс, зарегистрированный как upstream, а не плагин ContextForge. Документ в разделе 3 это и так допускает как альтернативу ("либо плагины ContextForge, либо отдельный MCP-сервер") — по итогам ревью кода для гос-контура однозначно стоит выбирать второй путь для всего, что не является тонкой, тривиальной и полностью подконтрольной логикой (фильтрация заголовков, инъекция заголовков и т.п., как в `header_filter`/`header_injector`).

## 5. Итоговая оценка по открытым вопросам раздела 10 ARCHITECTURE.md

| Вопрос из раздела 10 | Ответ по факту кода |
|---|---|
| Ложится ли ContextForge на per-user-сценарий с Tableau MCP | Частично из коробки (HTTP/SSE upstream — да, через `UpstreamSessionRegistry`); для stdio-апстриума Tableau MCP с динамическим PAT нужен эксперимент/доработка — документ прав, что без эксперимента решение неоднозначно |
| Tableau Server vs Cloud | Не влияет на применимость ContextForge как такового — это решается на уровне того, что регистрируется как upstream |
| Число одновременных пользователей | Архитектура ContextForge (Redis multi-cluster, session affinity) масштабируется штатно; стоимость на пользователя определяется тем, держите ли вы per-user upstream-процесс Tableau MCP (Вариант A) — за это отвечает не ContextForge, а схема обёртки Tableau MCP |
| Нужен ли весь набор тулов Tableau MCP | Не влияет на применимость ContextForge — независимо от объёма тулов, диспетчеризация и namespacing работают одинаково |
| Локальные модели как фолбэк | Не требует изменений в ContextForge — закрывается стандартным MCP-протоколом и Путём 2 (раздел 7) |

## 6. Рекомендации

1. Перед коммитом в ContextForge — обязательно провести эксперимент именно с реальным Tableau MCP (stdio) и динамическим PAT, как и предлагает сам документ; код подтверждает, что готового решения "из коробки" для этого конкретного кейса нет, есть только строительные блоки (`translate.py`, `UpstreamSessionRegistry`).
2. Не реализовывать PAT-enrollment-vault (раздел 6) штатными средствами passthrough-заголовков или плагином `vault` "as is" — оба варианта по факту требуют, чтобы клиент каждый раз присылал токен сам. Нужен собственный тонкий слой (плагин или роутер-патч), который на этапе OAuth-логина один раз получает PAT, кладёт в зашифрованную БД-таблицу по `sub`, и сам инжектит `X-Upstream-Authorization` перед вызовом апстрима — без участия клиента в передаче сырого PAT после enrollment.
3. Обязательно включить `MCPGATEWAY_UI_AIRGAPPED=true` и собрать образ через `scripts/download-cdn-assets.sh`; проверить итоговый Admin UI без доступа в интернет до приёмки.
4. `AUTH_ENCRYPTION_SECRET` завести через корпоративный секрет-менеджмент/HSM, не как plain env var в compose; формально зафиксировать процедуру ротации (в коде ротации нет — это операционная процедура).
